# Filled-In Examples

> See the seven governance templates in action, for two different contexts.

Blank templates tell you what to fill in. These examples show you what they look like when filled in — with real projects, real rules, real failure episodes, and real decisions.

---

## The Examples

### [Freelance Consultant — Maya Chen](freelance-consultant/)

A strategy consultant managing three simultaneous client projects (B2B SaaS, non-profit, startup). Month 5 of collaboration.

**Key features:**
- Codename system preventing real client names in deliverables
- Three projects with different rates (€950–€1,200/day) and deadlines
- Two documented failures: wrong client name in deliverable (F4) and unsourced benchmark (F3/F7)
- `/switch` command for explicit project context changes
- Revenue tracking against annual target

**Best for:** Freelancers, consultants, or anyone managing multiple clients or projects where context separation is critical.

---

### [PhD Researcher — Dr. Zara Ali](phd-researcher/)

A postdoctoral sociologist writing a 90,000-word monograph while managing journal submissions and funding applications. Month 8 of collaboration.

**Key features:**
- Absolute citation prohibition — created after Claude fabricated a Mouffe reference in month 1
- Evidential language levels (suggest/indicate/demonstrate) with rules for each
- Three documented failures with root causes, repairs, and rule changes
- Word count tracking by chapter and section against targets
- Read-only interview transcript folder with participant code system

**Best for:** Academic researchers, writers, or anyone where fabrication of sources carries serious professional consequences.

## How to Use These Examples

**If you're setting up for the first time:** Read the README for the example closest to your situation. Then use the actual template files to create your own versions — replace the example content with your context.

**If you're filling in a specific template:** Open the equivalent file in the example that best matches your work. Use it as a model for the kind of content that belongs in each section.

**If something has gone wrong:** Look at the FAILURE-RECOVERY.md in the relevant example. The documented failure episodes show the full cycle: what happened, why, the repair applied, and what changed.

---

## What All Examples Have In Common

Every example demonstrates the same core pattern:

1. **Running Document** = real project information, not placeholders
2. **Canonical Numbers** = actual figures Claude can reference without reconstructing
3. **Truth Protocol** = rules specific to the work context, not generic advice
4. **Failure Recovery** = real failures documented with root causes and fixes
5. **Rules evolve** — the rules in the Running Document were created after failures, not before them

The governance structure gets better through use. Start simple, document failures, add rules as you discover what needs protecting.
